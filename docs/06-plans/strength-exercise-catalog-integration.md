# Plan: Strength Exercise Catalog for Generated Workouts

## Goal

Use [hasaneyldrm/exercises-dataset](https://github.com/hasaneyldrm/exercises-dataset) as a canonical strength exercise catalog so AI-generated Gym / WeightTraining workouts produce `blocks.steps` that **reference stable catalog exercises**, instead of free-text names that are only fuzzy-matched afterward.

Secondary outcomes:

- A shared default exercise library athletes/coaches can browse and adopt
- Optional curated default Gym workout templates built from catalog IDs
- Richer enrichment (instructions, muscles, equipment) without requiring every user to build a library from scratch

---

## Current state

### Generation path

Strength structure is generated by the same Trigger tasks as endurance:

| Step              | Where                                                                                   |
| ----------------- | --------------------------------------------------------------------------------------- |
| Enqueue           | Planned workout / library template / chat tools / recommendation accept                 |
| Generate / adjust | `trigger/generate-structured-workout.ts`, `trigger/adjust-structured-workout.ts`        |
| Strength mode     | `legacy_json` when type matches `gym` or `weight`                                       |
| Prompt            | `trigger/utils/structure-generation-prompt.ts` → short `FOR STRENGTH` block             |
| Normalize         | `normalizeStructuredStrengthWorkout` in `server/utils/strength-exercise-library.ts`     |
| Library match     | `applyStrengthLibraryDefaultsToWorkout` in `server/utils/strength-exercise-matching.ts` |
| Validate          | `validateStrengthStructuredWorkout` (retry once on failure)                             |

### Strength structure shape

```text
structuredWorkout
├── blocks[]                    ← canonical
│   ├── type: warmup | single_exercise | superset | circuit | cooldown
│   └── steps[]                 ← one exercise per step
│       ├── name                ← free-text today
│       ├── libraryExerciseId?  ← per-user UUID when matched
│       ├── prescriptionMode / loadMode / defaultRest
│       └── setRows[]           ← per-set value / load / rest
└── exercises[]                 ← flattened compat projection
```

### What happens today

1. The model invents exercise **names** (library is not in the prompt).
2. Schema allows optional `libraryExerciseId`, but the model has no IDs to choose from.
3. After generation, we match against the user’s `StrengthExerciseLibraryItem` rows by exact ID → title → alias → optional LLM.
4. On match, we set `libraryExerciseId` and may fill video, notes, muscles, and weak prescriptions.

There is **no global/default catalog**. Empty user library ⇒ almost no enrichment. Matching is best-effort and fuzzy.

### Related models

| Model                         | Role                                                                       | Scope               |
| ----------------------------- | -------------------------------------------------------------------------- | ------------------- |
| `StrengthExerciseLibraryItem` | Planning library (title, aliases, muscles, prescription defaults, YouTube) | Per-user            |
| `WorkoutTemplate`             | Full session templates (`structuredWorkout` JSON)                          | Per-user            |
| `Exercise`                    | Completed-lift catalog from Hevy ingest                                    | Global, import-only |

## Recommended approach

Treat the dataset as a **system exercise catalog**, and teach generation to emit steps that reference it by stable ID.

```text
exercises-dataset (MIT metadata)
        │
        ▼
 SystemStrengthExerciseCatalog   ← new shared table / store
        │
        ├── Browse / search in exercise library UI
        ├── Optional "Add to my library" → StrengthExerciseLibraryItem
        └── Inject filtered catalog into strength structure generation
                    │
                    ▼
         blocks[].steps[] with catalogExerciseId (+ name)
                    │
                    ▼
         Resolve → enrich instructions / muscles / defaults
         Prefer user library overrides when present
```

**Why this over “clone 1,324 rows per user”:** one source of truth, updateable, works for users with an empty library, and matches how steps already want a reference ID.

**Why inject into the prompt (not only post-hoc match):** today’s matcher is a safety net. For reliable “steps referring to this data,” the model should choose IDs up front.

## Design decisions

### 1. Identity on a step

Add a stable catalog reference on each strength step, e.g.:

| Field               | Meaning                                           |
| ------------------- | ------------------------------------------------- |
| `catalogExerciseId` | System catalog ID (dataset `"0042"` or our cuid)  |
| `libraryExerciseId` | Optional user/coach library override (existing)   |
| `name`              | Display name (from catalog or user customization) |

Resolution order when enriching:

1. `libraryExerciseId` (user/coach customizations win)
2. `catalogExerciseId` (system defaults)
3. Fuzzy name match against user library, then catalog (fallback)

### 2. Catalog storage

Prefer a **DB-backed system catalog** (seeded from `exercises.json`) over reading a giant JSON at request time:

- Queryable / filterable for UI and prompt packing
- Stable internal IDs even if upstream dataset renumbers
- Room for our mapping fields (`movementPattern`, curated aliases, enabled flag)

Keep upstream `sourceId` (`"0001"` …) for re-sync.

### 3. Media

**Phase 1 default: metadata only** (names, muscles, equipment, instructions).  
Defer GIF/thumbnail hosting until Gym visual licensing is explicitly cleared. User library can keep YouTube `videoUrl` for personal media.

### 4. Catalog size in prompts

Do not dump all 1,324 exercises into every generation call.

- Maintain a **curated core set** (~100–200) always eligible for generation
- Optionally expand with a filtered slice (equipment / body part / session intent)
- Full catalog remains searchable in the UI

### 5. Default workouts

Separate from catalog import: curate a small set of Gym `WorkoutTemplate`s whose steps already carry `catalogExerciseId`. Ship as system templates or clone-on-first-use.

## Field mapping

| Dataset                        | Coach Wattz target                                                            |
| ------------------------------ | ----------------------------------------------------------------------------- |
| `id`                           | `sourceId` on system catalog                                                  |
| `name`                         | `title` / step `name`                                                         |
| `target` + `secondary_muscles` | `targetMuscleGroups` (via normalization map)                                  |
| `equipment`                    | New catalog field (filter/prompt); not on user library today                  |
| `instructions.{locale}`        | Catalog instructions; step/library `notes` when adopted                       |
| `image` / `gif_url`            | Deferred (licensing)                                                          |
| —                              | `movementPattern`, `intent`, prescription defaults → curated or derived later |

Build an explicit **muscle / body-part taxonomy map** (`pectorals` → our tokens, etc.). Do not pass raw dataset labels into UI filters unchanged.

## Required steps

### Phase 0 — Decisions & compliance

1. Confirm media policy: metadata-only for v1 vs pursue Gym visual license.
2. Confirm ownership model: system catalog visible to all vs clone-on-adopt only.
3. Confirm ID field name: `catalogExerciseId` vs generalizing `libraryExerciseId`.
4. Confirm i18n strategy for instructions (locale fallback chain).

### Phase 1 — Ingest system catalog

1. Add a system catalog model (or equivalent store) with at least:
   - internal `id`, `source` (`exercises-dataset`), `sourceId`
   - `title`, `aliases[]`
   - `bodyPart`, `equipment`, `targetMuscleGroups[]`
   - `instructions` (JSON by locale or primary + fallback)
   - `enabled`, `curatedForGeneration` (bool)
   - optional later: `movementPattern`, media URLs, attribution
2. Write an import/seed script that loads `data/exercises.json`, applies taxonomy mapping, and upserts by `source` + `sourceId`.
3. Mark a curated core subset `curatedForGeneration = true` (manual list or heuristic: common barbell/dumbbell/bodyweight compounds + key accessories).
4. Add a simple admin or script path to re-run import without duplicating rows.

### Phase 2 — Surface catalog in the exercise library UX

1. Extend library UI/API to list system catalog items (search, body part, equipment filters).
2. Support **Adopt**: copy/link a catalog item into `StrengthExerciseLibraryItem` (user can then edit aliases, prescription defaults, YouTube).
3. Keep user customs distinct from system rows (badges / sections).
4. Ensure coach ↔ athlete library access rules still apply only to user-owned items.

### Phase 3 — Wire generation to catalog IDs

1. Extend strength step types / AI JSON schema with `catalogExerciseId` (and keep `libraryExerciseId`).
2. In `generate-structured-workout` / `adjust-structured-workout`:
   - Load curated (+ optionally filtered) catalog slice for the user/session
   - Inject compact list into the strength prompt: `id`, `name`, `equipment`, `target`
   - Instruct the model to set `catalogExerciseId` for every work step when possible; invent names only for rare/custom moves
3. Update `applyStrengthLibraryDefaultsToWorkout` (or sibling resolver) to:
   - resolve `catalogExerciseId` first when no user library match
   - fill muscles / instructions / weak prescriptions from catalog
   - still prefer matched user library items when present
4. Strengthen `FOR STRENGTH` prompt bullets (ID-first, loaded lifts vs duration holds, block types).
5. Update validation: warn/retry when many steps lack both catalog and library IDs (soft at first, stricter later).
6. Ensure adjust path preserves IDs unless the user asks to replace exercises.

### Phase 4 — Default Gym workout templates (optional but high product value)

1. Curate ~8–15 starter templates (e.g. full body A/B, upper/lower, PPL, home bodyweight).
2. Persist as system templates or clone into the user’s workout library on first Gym use.
3. Every template step must carry `catalogExerciseId` (and sensible `setRows`).
4. Verify templates round-trip through normalize → editor → planned workout.

### Phase 5 — Hardening & ops

1. Tests: import mapping, ID resolution order, generation schema accepts `catalogExerciseId`, normalize preserves IDs, matching fallback still works.
2. Observability: log match rates (`catalog` vs `library` vs `unmatched`) in generate/adjust stages.
3. Document re-sync process when upstream dataset updates.
4. Product copy / attribution if any media is later enabled (`© Gym visual`).
5. Revisit related issues: `docs/issues/259-strength-normalization-retains-steps.md`, `docs/issues/261-strength-library-matching-can-fail-trigger.md` (matching must stay best-effort and budgeted).

## Out of scope (for this plan)

- Replacing Hevy’s completed-lift `Exercise` table
- Auto-generating entire periodized strength programs from the dataset alone
- Shipping Gym visual GIFs without a cleared license
- Migrating endurance interval `steps` grammar (unrelated)

---

## Risks

| Risk                                                 | Mitigation                                                                                                  |
| ---------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| Media license                                        | Metadata-only in v1; attribution + license if media added later                                             |
| Prompt token bloat                                   | Curated core + filtered slices only                                                                         |
| Taxonomy mismatch                                    | Explicit mapping table; curated aliases for common lifts                                                    |
| Three exercise silos (Hevy / user library / catalog) | Clear resolution order; stable `source` + `sourceId`; don’t merge Hevy completed rows into planning catalog |
| Model ignores IDs                                    | Soft validation + post-hoc catalog match as fallback                                                        |
| Matching cost / task failures                        | Keep enrichment best-effort with time budget (see issue 261)                                                |

---

## Success criteria

- Generated Gym workouts have `catalogExerciseId` (or resolved library ID) on the majority of work steps.
- Users with an empty personal library still get named, enrichable exercises from the system catalog.
- Exercise library UI can browse/search the catalog and adopt items.
- (Optional) New users can start from curated Gym templates whose steps already reference the catalog.
- No regression to endurance structure generation or existing user library CRUD.

## Suggested implementation order

```text
Phase 0 (decisions)
    → Phase 1 (catalog ingest)
        → Phase 2 (library UX)     ⎫ can overlap once ingest exists
        → Phase 3 (generation)     ⎭
            → Phase 4 (default templates)
                → Phase 5 (hardening)
```

Minimum useful slice: **Phase 0 + 1 + 3** (catalog exists and generation references it).  
Phases 2 and 4 make it feel like a product feature rather than only an AI improvement.
